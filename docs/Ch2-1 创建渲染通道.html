Ch2-1 创建渲染通道¶

好了，让我们正式进入EasyVulkan的旅途吧。
首先来新建两个文件：
一个叫 VKBase+.h ，另一个叫 EasyVulkan.hpp 。
在 VKBase+.h 中添加以下内容：
#pragma once
#include "VKBase.h"

namespace vulkan {
    /*待填充*/
}
在 EasyVulkan.hpp 添加以下内容：
#include "VKBase+.h"

namespace easyVulkan {
    using namespace vulkan;
    struct renderPassWithFramebuffers {
            renderPass renderPass;
            std::vector<framebuffer> framebuffers;
    };

    renderPassWithFramebuffers rpwf_screen;
    void CreateRpwf_Screen() {
    /*待填充*/
    }
}
VKBase.h 中我们对Vulkan对象做了最基础的封装，而 VKBase+.h 会被用于封住一些更复杂但经常发生的过程，这节课里我们暂时还不会写 VKBase+.h 。
EasyVulkan.hpp 会用来写我们的教程示例中特定情境相关的代码 —— 虽然这么说，其实还是有很大通用性的，我们最初的一个示例将会是一个简单的直接渲染到屏幕的渲染通道，对于简单的2D渲染来说可谓千篇一律，而我们封装完之后你就可以在不同的hpp中反复使用了。
renderPassWithFramebuffers 中包含了一个渲染通道和多个帧缓冲，因为我们得为交换链中的每一张图像建立一个帧缓冲。
我们的第一个 renderPassWithFramebuffers 对象叫做 rpwf_screen ，因为实在太长了，我终于用缩写了。
创建渲染通道的步骤有：

  • 为图像附件书写描述（VkAttachmentDescription）以及引用（ VkAttachmentReference，非C++语法上的引用）

  • 为子通道书写描述，以及子通道依赖

  • 创建渲染通道

之后我们会创建帧缓冲，并为应对可能发生的交换链重建（即窗口大小改变）设置回调。

图像附件的描述以及引用¶

来看下 VkAttachmentDescription 的成员：
struct VkAttachmentDescription {
VkAttachmentDescriptionFlags flags;
VkFormat format; //附件的图像格式
VkSampleCountFlagBits samples; //附件的采样次数
VkAttachmentLoadOp loadOp; //读取附件时对颜色和深度的操作
VkAttachmentStoreOp storeOp; //存储附件时对颜色和深度的操作
VkAttachmentLoadOp stencilLoadOp; //读取附件时对模板值的操作
VkAttachmentStoreOp stencilStoreOp; //存储附件时对模板值的操作
VkImageLayout initialLayout; //渲染通道开始前的内存布局
VkImageLayout finalLayout; //渲染通道结束时的内存布局
};
深度和模板值涉及到深度测试和模板测试，我们这次的示例只使用交换链中的图像，不附带深度和模板的附件。
让我们来看看代码吧：
VkAttachmentDescription attachmentDescription{};
attachmentDescription.format = graphicsBase::Base().SwapchainCreateInfo().imageFormat;//当然是交换链的图像格式
attachmentDescription.samples = VK_SAMPLE_COUNT_1_BIT;//在这个最简单的例子中，我们不使用多重采样（也就是常听闻的MSAA抗锯齿）
attachmentDescription.loadOp = VK_ATTACHMENT_LOAD_OP_CLEAR;//在读取图像时清空图像，你还可以选择LOAD或DONT_CARE
attachmentDescription.storeOp = VK_ATTACHMENT_STORE_OP_STORE;//在存储时存储，你还可以选择DONT_CARE
attachmentDescription.initialLayout = VK_IMAGE_LAYOUT_UNDEFINED;//内存布局为未定义
attachmentDescription.finalLayout = VK_IMAGE_LAYOUT_PRESENT_SRC_KHR;//顾名思义该布局用于呈现
我们只需要一个附件描述，因为对每张交换链中图像的描述都是一致的，不会有什么不同。
我们在读取图像时清空图像 —— 对于交换链中的图像，这是肯定的。如果你要做绘图工具（会保留之前所画的内容），你应该用会在之后课程中教授的离屏渲染，而不是直接画上屏幕。
由于我们压根没有模板值，所以不关心模板值的读取和存储。
初始内存布局为VK_IMAGE_LAYOUT_UNDEFINED可能导致上一帧的内容被丢弃，不过我们都清空了，还在乎上一帧的内容干嘛！
关于内存布局的转换会在子通道依赖后再讲述。
VkAttachmentReference 则简单很多，直接看具体示例：
VkAttachmentReference attachmentReference = { 0, VK_IMAGE_LAYOUT_COLOR_ATTACHMENT_OPTIMAL };
第一个字段代表附件的索引，如果我们有多个附件，那么附件描述会采用数组，而这里第一个字段即代表数组中的索引号。
由于我们只有一个附件描述，即每个帧缓冲只有一张图像，所以索引号为0。
第二个字段 layout 表示在渲染通道开始后应有的内存布局，VK_IMAGE_LAYOUT_COLOR_ATTACHMENT_OPTIMAL表示最适用于颜色附件的布局。
附件描述会用在之后渲染通道的创建信息中，而附件引用会用在子通道的描述中。

子通道描述以及子通道依赖¶

子通道描述¶

VkSubpassDescription 的内容：
struct VkSubpassDescription {
VkSubpassDescriptionFlags flags;
VkPipelineBindPoint pipelineBindPoint;
uint32_t inputAttachmentCount;
const VkAttachmentReference * pInputAttachments;
uint32_t colorAttachmentCount;
const VkAttachmentReference * pColorAttachments;
const VkAttachmentReference * pResolveAttachments;
const VkAttachmentReference * pDepthStencilAttachment;
uint32_t preserveAttachmentCount;
const uint32_t * pPreserveAttachments;
};
pipelineBindPoint 决定了这条子通道会使用何种管线，有VK_PIPELINE_BIND_POINT_GRAPHICS和VK_PIPELINE_BIND_POINT_COMPUTE以及VK_PIPELINE_BIND_POINT_RAY_TRACING_KHR可选。
inputAttachment 指的是输入附件，输入附件是用于采样的，并且只能获取位于同一像素位置上的颜色。这个教程直到延迟渲染才会使用输入附件。
colorAttachment 是最一般的颜色附件，用于让片段着色器写入颜色。
resolveAttachment 用于在多重采样下解析其他的附件（颜色以及深度模板），合成一张用于呈现的图像。
depthStencilAttachment 是深度与模板附件，深度与模板值是保存在在一张图像上的（也有可能只有深度值或模板值），用于让片段着色器写入深度和模板值。
preserveAttachment 是保留的附件，这类附件的内容不会在该子通道中被使用，并且其中的内容必须在该子通道的命令执行期间被保留 —— 程序会确保内容被保留，同时你自己也不应该在其他并发执行的命令中去修改他←应该没人会干这种事。
于是来看看具体代码：
VkSubpassDescription subpassDescription{};
subpassDescription.pipelineBindPoint = VK_PIPELINE_BIND_POINT_GRAPHICS;
subpassDescription.colorAttachmentCount = 1;
subpassDescription.pColorAttachments = &attachmentReference;
我们只有一个颜色附件，于是就非常简单地写完了。

子通道依赖¶

VkSubpassDependency 的内容：
struct VkSubpassDependency {
uint32_t srcSubpass;
uint32_t dstSubpass;
VkPipelineStageFlags srcStageMask;
VkPipelineStageFlags dstStageMask;
VkAccessFlags srcAccessMask;
VkAccessFlags dstAccessMask;
VkDependencyFlags dependencyFlags;
};
首先引入概念：
第一同步域：对于子通道依赖中的情况，指代 srcSubpass 中， srcStageMask 所描述的管线阶段中 srcAccessMask 所表示的访问操作。
第二同步域：对于子通道依赖中的情况，指代 dstSubpass 中， dstStageMask 所描述的管线阶段中 dstAccessMask 所表示的访问操作。
VkSubpassDependency 是一种同步手段，它会确保第一同步域中的操作（及其之前其他管线阶段的操作）在第二同步域中的操作（及其之后其他管线阶段的操作）之前完成，并且附件的内存布局转换会在两者之间发生。
srcSupass 和 dstSubpass 处填入的都是子通道在整个渲染通道中的索引，有一个特殊值VK_SUBPASS_EXTERNAL表示渲染通道外部的操作，用于 srcSubpass 时表示本渲染通道开始前的操作，用于 dstSubpass 时表示本渲染通道结束后的操作。
srcSupass 同 dstSubpass 一致时，称该子通道为自依赖的。
因为是mask，所以你可以用位或指定多于一个阶段以及多于一种访问操作。
另外，你要注意的是，有些阶段只存在于特定类型的渲染管线，以及不同的阶段所能使用的访问操作是不同的，具体参考官方标准：
https://vulkan.lunarg.com/doc/view/1.2.148.1/linux/chunked_spec/chap7.html#synchronization-pipeline-stages
如果在srcStageMask上使用VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT，说明第二同步域的所有操作必须等待先前的命令全部完成。
在dstStageMask上使用VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT，说明之后的所有命令都得在第一同步域的操作完成后执行。
dependencyFlags 在1.0版本中可用的值有VK_DEPENDENCY_BY_REGION_BIT，它表示这个子通道依赖可以被分为多个frambuffer-local依赖。
BY_REGION顾名思义就是按区域，所以frambuffer-local依赖即是按区域的依赖。至于这个区域多大你是不清楚的，这取决于具体的硬件，你应该假设可能的最小区域 —— 像素或采样点。
构成frambuffer-local依赖的条件之一在于，你能确保第二同步域中的操作在访问图像的某个像素或采样点时，第一同步域中对该像素或采样点的操作已经完成了。
另一个条件是你的srcStageMask或dstStageMask中包含了以下四个阶段之中的至少一个（按发生的顺序排列）：
VK_PIPELINE_STAGE_EARLY_FRAGMENT_TESTS_BIT
VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT
VK_PIPELINE_STAGE_LATE_FRAGMENT_TESTS_BIT
VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT
这些阶段统称为framebuffer-space阶段，如果没有包含以上阶段，那么这个依赖无所谓frambuffer-local或frambuffer-global，此时写上VK_DEPENDENCY_BY_REGION_BIT没有任何效果。
VK_DEPENDENCY_BY_REGION_BIT是一个聊胜于无的flag，绝大多数情况下你都可以使用它，并且很多硬件在使用该flag时有更好的表现。
什么情况下不该使用使用？举例而言，如果你的前一个子通道在写入颜色附件的某个像素A时，同时写入了并非以附件来使用的内存区域（比如写入Storage Buffer）的某个位置C，之后你在另一个子通道中继续处理像素A，并且访问了前述内存区域的位置D，而位置D本该在前一个子通道处理像素B时写入 —— 于是对应关系就被打破了。如果依赖按区域发生，你无法确保B在前一个子通道里被处理完了，也就无法确保D已经被写入。
此外，如果子通道为自依赖，且 srcAccessMask 和 dstAccessMask 都包括了framebuffer-space阶段，则必须使用这个flag。
最后还有一个问题，什么情况下需要写子通道依赖？
这个问题比较复杂，在回答这个问题之前，我们先来看看接下来的代码：
VkSubpassDependency subpassDependency{};
subpassDependency.srcSubpass = VK_SUBPASS_EXTERNAL;
subpassDependency.dstSubpass = 0;
subpassDependency.srcStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
subpassDependency.dstStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
subpassDependency.srcAccessMask = VK_ACCESS_COLOR_ATTACHMENT_READ_BIT;
subpassDependency.dstAccessMask = VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT;
subpassDependency.dependencyFlags = VK_DEPENDENCY_BY_REGION_BIT;
在COLOR_ATTACHMENT_OUTPUT阶段，混色结果被输出至颜色附件，若使用了多重采样，还会包括对多重采样附件的解析。
COLOR_ATTACHMENT_READ和COLOR_ATTACHMENT_WRITE是对颜色附件的读写操作，只能在COLOR_ATTACHMENT_OUTPUT阶段使用。
这个子通道依赖的意思是：0号子通道中对颜色附件的写入操作，需在外部对该颜色附件的读取操作全部完成后才能继续。
外部对该颜色附件的读取操作是指什么？
一个通常的Rendering Loop是： 获取下一张图像 —— 渲染图像 —— 呈现图像。
所谓下一张图像是因为有交换链中有好几张图像会进行交换。
读取操作指的是在前一次循环中呈现图像，图像必须被呈现引擎读取完了，才能安全地进行内存布局转换，然后被用于写入。
至于为什么我们要写这个子通道依赖？
事实上，哪怕你不写这个子通道依赖，渲染通道也是有两个隐式子通道依赖的，一个发生在渲染通道开始时，一个发生在渲染通道结束后。
开始时的子通道依赖将附件的内存布局从 VkAttachmentDescription 中的 initialLayout 变为 VkAttachmentReference 中的 layout ，然后结束时的子通道依赖又会把布局从 VkAttachmentReference 中的 layout 变为 VkAttachmentDescription 中写 finalLayout 。
我们写的这个子通道依赖覆盖了在渲染通道开始时的隐式子通道依赖。
覆盖开始时依赖的条件：srcSubpass是VK_SUBPASS_EXTERNAL，dstSubpass是第一个有附件的子通道。
相应地，srcSubpass是最后一个有附件的子通道，dstSubpass是VK_SUBPASS_EXTERNAL时会覆盖渲染通道结束时的依赖。
而获取图像到执行命令缓冲区的步骤，按1.0的标准所言，是这样的：
1.通过 vkAcquireNextImage 获取图像，并附带一个semaphore，待成功获取完图像后点亮semaphore
2.在提交命令缓冲区时，你会填写一个提交信息 VkSubmitInfo ，填入这个semaphore以及相应的等待阶段VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT。
VkSubmitInfo 中有一个成员叫pWaitSemaphores用于传入需要等待的semaphore数组，1.0至今（是的至今没改）的官方标准上是如此描述它的：
pWaitSemaphores is a pointer to an array of VkSemaphore handles upon which to wait before the command buffers for this batch begin execution.
3.在命令缓冲区开始执行前，必须等待之前的命令到达VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT并检查semaphore，待semaphore点亮后才能执行命令。
而 vkCmdBeginRenderPass 是记录在命令缓冲区中的命令，隐式的子通道依赖又是在渲染通道开始时执行。
仔细一想，是否可以推理得出无论如何内存布局转换都会在呈现结束，获取到图像之后发生呢？
严格按照1.0的标准来讲当然是这样啦！既然如此自己显式地写一个子通道依赖似乎完全没必要吧！？
然而，在之后的标准中又增加了一条注释：
（出现在chap7.html#synchronization-semaphores-waiting）
Some implementations may be able to execute transfer operations and/or vertex processing work before the semaphore is signaled.
中文：有些Vulkan实现可能会在信号灯被点亮前开始执行转移，或开始处理顶点。
再简单点：有些Vulkan实现可能不会等待信号灯被点亮，就开始执行命令缓冲区。
这就是为什么我们要显式地定义上文的子通道依赖 —— 当渲染通道开始时，之前的操作可能并未完成，而内存布局转换最迟可以到输出颜色之前。
（这个问题纠结了笔者好久。。。）

创建渲染通道¶

VkRenderPassCreateInfo 的内容：
struct VkRenderPassCreateInfo {
VkStructureType sType;
const void * pNext;
VkRenderPassCreateFlags flags;
uint32_t attachmentCount;
const VkAttachmentDescription * pAttachments;
uint32_t subpassCount;
const VkSubpassDescription * pSubpasses;
uint32_t dependencyCount;
const VkSubpassDependency * pDependencies;
};
没什么需要说明的，填入附件描述、子通道描述、子通道依赖就完事。
我会用在Ch1-4中写的renderPass类来创建一个渲染通道，所以以下代码中sType就直接省略了：
VkRenderPassCreateInfo renderPassCreateInfo{};
renderPassCreateInfo.attachmentCount = 1;
renderPassCreateInfo.pAttachments = &attachmentDescription;
renderPassCreateInfo.subpassCount = 1;
renderPassCreateInfo.pSubpasses = &subpassDescription;
renderPassCreateInfo.dependencyCount = 1;
renderPassCreateInfo.pDependencies = &subpassDependency;
rpwf_screen.renderPass.Create(renderPassCreateInfo);

创建帧缓冲¶

首先我们定义两个lambda表达式，之后需要将它们push进 graphicsBase 中创建/销毁交换链的回调函数vector里。
因为销毁帧缓冲的方式很简单，所以这里已经写好了。
auto CreateFramebuffers = [] {
    /*待填充*/
};
auto DestroyFramebuffers = [] {
    //clear掉vector时自动调用每个对象的析构器。如果你没有仔细看前一节的代码，请现在去读一下renderPass的析构器。
    rpwf_screen.framebuffers.clear();
};
又来了（我甚至觉得我有必要写么？）， VkFramebufferCreateInfo 的内容：
struct VkFramebufferCreateInfo {
VkStructureType sType;
const void * pNext;
VkFramebufferCreateFlags flags;
VkRenderPass renderPass;
uint32_t attachmentCount;
const VkImageView * pAttachments;
uint32_t width;
uint32_t height;
uint32_t layers;
};
仍旧无须解释。
我们之前存储了交换链的创建信息，可以从中获取 width 和 height
一般而言（我也不知道有什么不一般的情况），交换链图像的大小即是窗口大小，并且我们会经常需要取得这个大小，所以我们在 graphicsBase 下面加入以下代码：
constexpr const VkExtent2D& windowSize = graphicsBase::Base().SwapchainCreateInfo().imageExtent;
填充 CreateFramebuffers ：
auto CreateFramebuffers = [] {
    rpwf_screen.framebuffers.resize(graphicsBase::Base().SwapchainImageCount());//为每一张图像创建帧缓冲
    VkFramebufferCreateInfo framebufferCreateInfo{};
    framebufferCreateInfo.renderPass = rpwf_screen.renderPass;
    framebufferCreateInfo.attachmentCount = 1;//我们只有一个附件
    framebufferCreateInfo.width = windowSize.width;
    framebufferCreateInfo.height = windowSize.height;
    framebufferCreateInfo.layers = 1;//当然是1啦
    for (size_t i = 0; i < graphicsBase::Base().SwapchainImageCount(); i++)
    {
        VkImageView attachment = graphicsBase::Base().SwapchainImageView(i);//记得我们之前创建了image view么？
        framebufferCreateInfo.pAttachments = &attachment;
        rpwf_screen.framebuffers[i].Create(framebufferCreateInfo);//这个函数会调用 vkCreateFramebuffer
    }
};
推入vector并执行 CreateFramebuffers ：
graphicsBase::Base().PushCallback_CreateSwapchain(CreateFramebuffers);
graphicsBase::Base().PushCallback_DestroySwapchain(DestroyFramebuffers);
CreateFramebuffers();
最后我们在主函数中，在初始化窗口之后，调用 CreateRpwf_Screen 。
